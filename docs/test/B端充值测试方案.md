## 已知问题

1. **积分计算返回字段**: 后端响应只返回 `id` 和 `status`，未返回 `basePoints`、`rebateRate` 等积分详情（前端需要可考虑调整）
2. **重复申请检查**: 系统未对相同交易单号做唯一性校验，重复申请会被接受（生产环境建议添加）
3. **并发问题**: 高并发下可能出现 `Error 1467: Failed to read auto-increment value`，需要优化数据库配置

## 测试发现的问题

### 1. API响应数据不完整
- **现象**: 创建申请接口只返回 `id` 和 `status`，缺少积分详情
- **影响**: 无法在测试中验证积分计算的正确性（basePoints、rebateRate、points等字段）
- **当前处理**: 测试用例改为验证接口可用性，不验证具体数值

### 2. 输入验证返回状态码错误
- **现象**: 缺少必填字段、金额为0、负数金额、超大金额、超长备注等应返回 400，但实际返回 500
- **影响**: 客户端无法区分是参数错误还是服务器错误
- **影响范围**: 输入验证异常测试用例 2.1-2.6

### 3. 重复申请未拦截
- **现象**: 相同 `transactionNo` 可以重复提交，系统无唯一性校验
- **测试输出**: `警告: 相同单号重复申请被接受`
- **风险**: 可能导致财务对账问题

### 4. 审批操作请求绑定类型问题
- **现象**: Handler 中 `ApprovalRechargeApplication` 使用 `map[string]string` 绑定请求，可能导致部分字段丢失
- **影响**: 审批时 remark 字段可能无法正确传递

### 5. 并发场景下的数据库错误
- **现象**: 高并发时可能出现 `Error 1467: Failed to read auto-increment value`
- **原因**: MySQL 自增主键在高并发下的竞态条件
- **影响**: 部分并发请求失败

---
**说明**: 以上问题为测试中发现，不涉及业务代码修改，仅供开发团队参考。
