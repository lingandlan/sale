# 太积堂充值与门店管理系统 - 开发实现文档

## v1.1 | 2026-04-08

---

## 一、系统概述

### 1.1 项目定位

本系统是太积堂的充值与门店管理平台，核心解决"充值账务清晰、门店卡可管控、操作可追溯"的问题。

两大系统并列独立部署，通过 `user_id` 数据互通。

### 1.2 与商城系统关系

| 系统 | 定位 | 数据互通 |
|------|------|----------|
| 商城系统 | 会员体系、佣金计算、成组成团、积分兑换、会员升级 | 通过 user_id 识别同一用户 |
| 本系统 | 充值中心管理、充值记录、门店卡发行核销、会员管理、操作日志、C端小程序 | 通过 user_id 向商城查询消费记录、通知充值 |

### 1.3 核心功能模块

```
┌─────────────────────────────────────────────────────────┐
│                     充值与门店管理系统                      │
├─────────────────────────────────────────────────────────┤
│  充值中心管理  │  门店管理  │  B端充值  │  C端充值       │
├─────────────────────────────────────────────────────────┤
│  门店卡管理   │  操作日志   │  操作员管理  │  C端小程序    │
└─────────────────────────────────────────────────────────┘
```

### 1.4 角色定义

| 角色 | 说明 | 操作范围 |
|------|------|----------|
| 超管 | 系统最高权限 | 所有操作；B/C端充值审批；充值中心/门店管理；卡开店发放；权限管理 |
| 总部管理员 | 专门负责审批 | B端充值审批；C端充值审批；查看所有充值记录和操作日志 |
| 总公司财务运营 | 总公司财务人员 | 发起B端充值申请；查看所有记录；导出；卡入库 |
| 充值中心管理员 | 各充值中心财务负责人 | C端充值录入；卡发放/核销；管理本中心操作员；查看本中心所有日志 |
| 充值中心操作员 | 各充值中心一线操作人员 | C端充值录入；卡发放/核销；查询用户消费记录 |
| 持卡消费者 | C端小程序用户 | 查看持有的门店卡和消费记录 |

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | bigint | 是 | 主键 |
| recharge_center_id | bigint | 否 | 所属充值中心ID（超管可为空）|
| username | varchar(32) | 是 | 用户名 |

**角色说明**：
- 角色 0=超管（系统最高权限，`recharge_center_id` 可为空，拥有所有权限）
- 角色 1=充值中心管理员
- 角色 2=充值中心操作员
- 超管/总部管理员/总公司财务运营 通过 casbin_policy 控制 API 权限

**超管账号存储**：
- 统一存储在 `operator` 表，`role = 0` 表示超管
- 初始超管账号通过数据库迁移脚本（goose）插入
- 超管可以绑定充值中心（成为总部管理员或总公司财务运营），也可以不绑定

**卡状态机**：
```

              已消费完  解冻 → 已发放
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | bigint | 是 | 主键 |
| operator_id | bigint | 是 | 操作员ID |
| operator_type | varchar(32) | 是 | 操作员类型 |
| action | varchar(32) | 是 | 操作类型 |

| before_data | json | 否 | 变更前数据 |

**哈希链机制**：
- 每条日志的 `hash` = SHA256(id + operator_id + action + entity_id + before_data + after_data + prev_hash)
- 查询时校验 hash 链完整性

## 四、API 接口设计

### 4.1 接口规范

#### 认证方式

| 端 | 方式 | 说明 |
|---|------|------|
| 管理端 | 手机号+密码 | JWT Access Token（24h）+ Refresh Token（7d）|
| C端小程序 | 手机号+验证码 | JWT Access Token |

#### 请求格式

```
Content-Type: application/json
Authorization: Bearer <token>
```

#### 响应格式

```json
// 成功
{
    "code": 0,
    "message": "success",
    "data": {}
}

// 错误
{
    "code": 400,
    "message": "参数错误",
    "detail": "具体错误信息"
}
```

#### 状态码

| 状态码 | 说明 |
|--------|------|
| 0 | 成功 |
| 400 | 参数错误 |
| 401 | 未认证 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 409 | 资源冲突 |
| 500 | 服务器错误 |

## 六、技术决策

### 6.1 积分存储

- 使用 `decimal(20,2)` 存储积分，保证精度
- 所有积分计算在后端完成，前端只展示

### 6.2 充值余额计算

```
充值中心余额 = 累计B端充值积分 - 累计C端充值积分
```

### 6.3 卡号生成

- 格式：`前缀 + 8位序号`（如 `TJ12345678`）
- 前缀可配置
- 序号由系统自增生成

### 6.4 哈希链实现

```go
func calculateHash(log *OperationLog) string {
    data := fmt.Sprintf("%d|%d|%s|%d|%s|%s|%s",
        log.ID,
        log.OperatorID,
        log.Action,
        log.EntityID,
        string(log.BeforeData),
        string(log.AfterData),
        log.PrevHash,
    )
    return sha256.Sum256([]byte(data))
}
```

```go
// pkg/mall/client_mock.go

type MallClientMock struct {
```

#### 8.3.3 客户端工厂

```go

type MallClientInterface interface {
    GetAccessToken() error
```
